我们组用 Perforce，隔壁用 Git，
学校教的是 SVN…
这三个到底有啥区别？
为什么不统一用一个？🤔
好问题！这三个都是版本控制工具，
先想清楚版本管理要解决什么：
① 谁·在什么时候·改了哪一行
② 多人同时改，怎么合并不打架
③ 改错了，怎么一键回退到从前
三者就是针对这三点的三种答案！
① SVN — 集中式
只有一个中心服务器 📡
版本号统一递增，结构简单！
但断网就啥都干不了 😅

原来设计目标
根本不一样！
SVN
✔ 结构简单，心智负担低
✔ 目录、权限管理好
✘ 断网啥都干不了
✘ 分支合并体验差
Perforce
✔ 百万文件也流畅
✔ 签出锁定防冲突
✘ 商业授权贵
✘ 分支操作重
Git
✔ 离线可用
✔ 分支几乎零成本
✘ 仓库随历史变大
✘ 二进制文件弱
那我该怎么选呀？
好难决定… 😩
关键看文件类型
和团队协作模式！
不是哪个更先进，
是哪个更适合你 😎
🗺️ 选型决策速查
📝
以文本代码为主，需要频繁分支 & 开源协作
→ Git + GitHub
🎮
海量二进制大文件（游戏美术/芯片设计），需签出锁定
→ Perforce
🏢
小团队、文本代码，不需要复杂分支，心智负担低优先
→ SVN
🔀
代码 + 大二进制都有？可以代码用 Git、资产用 Perforce，分开管理各取所长
→ 混用
搞清楚设计逻辑，
遇到新项目该选什么、
老项目要不要迁移，
心里就有数了！💪
💡 三条实用建议：
① 大二进制别直接 git add，用 LFS
② 提交信息认真写，任何工具都一样
③ Perforce 改文件前记得 p4 edit！

版本历史放在哪里 → 决定了整套协作方式
SVN 留在中心 · Perforce 中心+锁定优化 · Git 分发到每个人
选型不问"哪个先进"，问"哪个适合你的场景"
