# 写事务的边界与拆分：Large Write 是什么

## 为什么写比读更复杂

*芯片验证 · PCIe · 总线协议 · 写事务*

---

**摘要**：在现代总线协议中，一笔写事务往往不能任意大。它受到边界对齐规则的约束，超过边界就必须拆分成多个子事务。如何拆、拆出的碎片如何串联、接收方如何重新识别完整的写——这套机制因 Large Write 特性的开启与否而截然不同。本文用四张图把这个过程讲清楚。

---

## 一、为什么要限制写事务的大小

总线上的写事务不是越大越好。发送方在发出写数据之前，需要确认接收方有足够的缓冲空间来接收这些数据——这就是信用（Credit）机制。信用以固定的粒度分配：发送方每消耗一定量的数据，就要消耗一个信用；信用耗尽就必须等待接收方释放。

写事务的最大尺寸与信用粒度直接挂钩。如果信用粒度是 64B，那一次写事务最多发 64B 才能保证一个信用就够；如果粒度是 256B，写事务可以放大到 256B。这是 Large Write 特性存在的最直接动机。

此外，地址对齐也有要求。一笔写事务不能随意跨越边界——超出边界必须拆分，避免接收方在单次操作中处理地址范围过大的写请求。

---

## 二、普通写模式：64B 边界 + Chain 串联

在不启用 Large Write 的情况下，写事务的边界是 **64B**。一笔原始写事务如果跨越了 64B 边界，就必须在边界处切开，变成多个片段。

![普通写模式：64B 边界拆分 + Chain 串联](large_write_boundary.png)

拆出来的多个片段不是完全独立的——它们通过 **chain 位**串联在一起：

- 非最后一个片段：chain=1，表示"后面还有，这不是完整的写"
- 最后一个片段：chain=0，表示"这笔写事务到这里结束了"

接收方看到 chain=1 时，会把当前片段先缓存，等到收到 chain=0 的片段后，才把整批数据合并成一次完整的写操作。

但这个机制有一个约束：**所有 chain 片段必须落在同一个 256B 窗口内**。跨越 256B 边界时，chain 必须重置为 0，下一个 256B 窗口内的片段重新开始。换句话说，每个 256B 窗口是一个独立的 chain 域。

---

## 三、Large Write 模式：256B 边界，去掉 Chain

启用 Large Write 后，写事务的边界从 64B 扩展到 **256B**。拆分逻辑变得简单很多：

![Large Write 模式：256B 边界，无 Chain](large_write_mode.png)

如果整笔写事务在 256B 窗口内，直接整体发送，chain=0，无需任何串联。

如果写事务跨越了 256B 边界，在边界处拆开，每个片段独立发送，各自 chain=0，接收方逐个处理即可。

没有 chain=1，没有"等待最后一片"的逻辑，接收方看到每笔写请求都是完整独立的。这大幅简化了接收端的处理流程，也减少了因 chain 带来的额外状态管理开销。

信用粒度同步从 64B 升级为 **256B**，一个信用能承载更多数据，信用消耗频率降低，总线效率更高。

---

## 四、读事务不受影响

无论 Large Write 是否开启，**读事务的行为是一样的**：始终按 256B 边界拆分，且读事务从来不需要 chain 机制——读请求只有地址和长度，没有数据负载，接收方可以立即识别每笔读请求的完整范围。

Large Write 这个名字容易误导人——它只影响写事务，对读事务没有任何作用。

---

## 五、普通模式 vs Large Write 模式：一眼看差异

![对比：普通写模式 vs Large Write 模式](large_write_compare.png)

两种模式的核心区别在于写边界和 Chain 机制。普通模式下，64B 是写事务的自然粒度，超出就需要串联；Large Write 模式下，256B 才是边界，每笔写请求完全独立，逻辑更清晰。

---

## 六、拆分决策的完整流程

![写事务拆分决策流程](large_write_flow.png)

收到写事务后，首先判断是否是 Atomic 操作——Atomic 不拆分，整体发送。然后判断整笔事务是否在写边界内（普通模式看 64B，Large Write 模式看 256B），在边界内则直接发送。超出边界后才进入拆分逻辑：Large Write 开启时，各片段独立发送；否则需要逐片判断是否到达 256B 窗口边界，未到达的片段 chain=1，到达或末尾的片段 chain=0。

---

## 七、总结

Large Write 解决的问题是：在保证地址对齐约束的前提下，允许更大的写事务单次传输，同时去掉 chain 串联机制，简化发送端和接收端的逻辑。

普通写模式的三个关键数字：64B（写边界）、chain=1/0（串联标志）、256B（chain 窗口上限）。

Large Write 模式的三个关键数字：256B（写边界）、chain=0（永远独立）、256B 信用粒度（更高效）。

读事务不在此讨论范围内，两种模式下读的边界和行为一致。
